# Framework decision

Start with the dominant constraint: interaction fidelity, deployment environment, privacy, latency, offline use, integration surface, data sensitivity, or evaluation rigor. State the smallest architecture that satisfies it, then name consequences and discarded alternatives.

Do not say “use <technology> because it is popular.” A framework is a consequence of the product and constraints, not a default. For agentic work, name the action space, actionable errors, budgets, and eval surface in product-readable terms.
